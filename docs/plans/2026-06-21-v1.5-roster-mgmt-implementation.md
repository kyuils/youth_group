# v1.5 구현 계획 — 교사 명단 관리 + 새가족 중복 정리 + 출석 UX 개선

작성일: 2026-06-21 · 상태: 초안(시니어 검토 대기)

## 요구사항(원문 → 라벨)

- **R1** 교사가 자신의 반에 학생을 추가할 수 있는 기능 추가
- **R2** 교사가 자신의 반 학생의 인적사항을 수정하는 기능을 **출석부 외부에 표시**
- **R3** 새가족부 윤현성이 최용수 교사반과 **중복**된 문제 해결 + **새가족부에서 윤현성 삭제**
- **R4** 출석부 필터/정렬 바 디자인 변경 — **이름순·학년순·최근출석순을 아래 줄로 이동**해 분류 시인성 향상
- **R5** **결석 체크 시 사유 입력을 선택(optional)으로** 변경 (체크 후 필요한 교사만 기재)

## 현재 구조(확인된 사실)

- FE: `web/index.html` 단일 React/Babel 파일. 탭: `roster(출석)/weekly(현황)/prayer(기도제목)/newcomers(새가족부)/admin(관리자)`.
- BE: `gas/*.gs` + Google Sheet. `STUDENTS`는 `2026 반편성`(RAW_CLASS)의 읽기전용 ArrayFormula 뷰. **쓰기는 RAW_CLASS에만**. 학생 반 = P열.
- 학생 수정 액션 `updateStudent` 이미 존재. 진입점은 출석탭 학생이름→`StudentDetail` 모달뿐.
- 교사가 일반 학생을 자기 반에 추가하는 액션 **없음**(`addNewcomer`만 있음).
- 결석 사유는 `EtcInputSheet`에서 **필수**.

## 설계

### R1 + R2 — 신규 "명단" 탭 (RosterManageView)

출석(체크) 흐름과 분리된 **명단 관리 전용 탭**을 신설해 R1(추가)·R2(수정 외부 노출)를 함께 해결.

- FE: 새 탭 `['manage','명단']`을 일반 교사·관리자에게 노출(출석 다음). 새가족부 전담 교사(본인 반 없음)에겐 미노출.
- 화면: 본인 반 명단(`getRoster`) 카드 리스트. 각 행 = 이름·학년·성별·연락처 + `[정보]`(→ `StudentDetail`, 기존 수정 버튼 재사용).
- 상단 `[＋ 학생 추가]` → `AddStudentSheet` 폼: 이름*(필수), 성별, 학년(드롭다운), 연락처, 생년월일, 학교, 부모님연락처, 주소, 비고.
- BE: 신규 액션 `addStudent` — RAW_CLASS에 행 추가(B=교사명, P=반=auth.teacher), id는 `getNextStudentId_`. 본인 반만(관리자는 `classFilter` 허용). 쓰기 후 `STUDENTS_MIN_v3` 캐시 무효화. `sanitizeCell_`로 수식 인젝션 방어.
- `StudentDetail`의 수정 기능은 그대로(새가족부 정보에서도 사용). 관리 홈은 새 명단 탭.

### R3 — 새가족 중복(윤현성)

3중 방어:

1. **구조적 dedup**: `handleGetNewcomers`의 활성(미등반) 목록에서, 같은 id가 `loadStudentsMin_()`에 **반≠새가족인 실제 학생으로 존재**하면 제외. → 배포 즉시 윤현성이 새가족부 활성 목록에서 사라짐. 정상 새가족(새가족부에만 존재)·등반자(graduatedOn 보유)는 영향 없음.
2. **삭제 기능 신설**: 액션 `deleteNewcomer`(새가족부 active=FALSE) + 새가족 카드에 `[삭제]` 버튼(admin/newcomer_staff). 향후 동일 문제 자가 해결.
3. **일회성 마이그레이션** `migration_cleanup_orphan_newcomers()` — 새가족부 행 중 id가 실제 학생으로도 존재하는 행을 비활성화. 윤현성 영구 정리.

> ⚠ 라이브 시트 쓰기는 이 환경에서 불가. 윤현성 데이터의 실제 정리는 **운영자가 GAS 재배포 + 마이그레이션 실행(또는 앱에서 삭제 클릭)** 해야 완료된다. 코드/마이그레이션은 본 작업에서 제공.

### R4 — 필터/정렬 바 2행 분리

- `RosterView`의 `.filter-bar`를 **2행**으로: 1행 = 필터(전체/미체크/⚠3주결석), 2행 = 정렬(이름순/학년순/최근출석순) + "정렬" 라벨. `.filter-sep` 제거.
- CSS: `.filter-bar`를 컬럼 컨테이너로, 각 행 `.filter-row`(flex, wrap). 칩 스타일 유지.
- 일관성: `WeeklyView`의 정렬 바도 동일 톤 검토(과확장 주의 — 우선 출석탭에 집중).

### R5 — 결석 사유 선택화

- `setStatus`에서 ABSENT는 모달 없이 **즉시 저장**(`saveAttendance(id,'ABSENT','')`).
- 카드에 `cur==='ABSENT'`일 때만 `[✎ 사유]` 보조 버튼 노출 → `EtcInputSheet`(선택) 열기. 기존 사유는 `attRows`의 `etcText`로 프리필.
- 일괄(ABSENT)도 사유 강제 없이 저장.
- `EtcInputSheet`: ABSENT `required=false`, 안내문구/별표 수정. ETC(외부예배)는 기존 유지(모달·선택).
- 문서: `docs/ops/09-v1.4-deploy.md`의 "(#2) 결석 사유 필수" 항목을 v1.5에서 선택으로 변경됨으로 갱신.

## 시니어 검토 반영 (확정 결정)

- **[블로킹 수정] R3 마이그레이션 가드**: `migration_cleanup_orphan_newcomers()`는 단순히 "실제 학생으로 존재하는 id"를 비활성화하면 **정상 등반자까지 삭제**한다(등반자도 RAW_CLASS에 반=교사로 존재). 조건을 `(id가 실제 학생, 반≠새가족) AND (등반일 비어있음)`으로 한정한다. 윤현성(등반일 없음)만 정리되고 등반자는 보존.
- **R3 핵심은 구조적 dedup**: `handleGetNewcomers`의 활성 목록 필터가 load-bearing 수정. 활성 목록은 `!graduatedOn`만 보므로 등반자는 애초에 진입 안 함 → **dedup은 자동으로 등반자-안전**. 배포 즉시 윤현성이 사라짐, 시트 조작 불필요. 판정은 기존 `findStudentMinById_(id)['반'] !== '새가족'` 재사용(pre-v1.4 새가족의 원본 반='새가족' 행을 올바르게 보호). 마이그레이션/삭제UI는 보조.
- **R1 append 안전**: RAW_CLASS는 `getLastRow()`가 팬텀행으로 불안정 → 절대 naive `appendRow` 금지. `copyNewcomerToRawSheet_`의 C열 상향 스캔 로직을 공유 헬퍼 `appendRawClassRow_(values)`로 추출해 두 경로가 어긋나지 않게. B=교사명·P=반 모두 설정.
- **R5 EtcInputSheet 공유 주의**: `NewcomersView`도 같은 시트를 ABSENT에 사용. 변경 시 **두 진입점 모두** 테스트. RosterView=1탭 즉시저장+`[✎ 사유]` 보조버튼(attRows의 etcText 프리필). NewcomersView=모달 유지하되 선택(required=false). ETC는 불변.
- **R2**: 출석탭의 이름→상세 진입은 **유지**(편의 회귀 방지). 명단탭이 "출석부 외부" 노출을 충족.
- **R4**: 출석탭 바만 2행화. WeeklyView는 정렬 셋이 달라 건드리지 않음(과확장 금지).
- **탭 6개(관리자)**: `.topnav` 가로 스크롤이라 허용.
- **배포**: `docs/ops/10-v1.5-deploy.md` 신규 작성 + `BUILD_TAG`를 `v1.5.0-...`로 올림(운영자가 `doGet`으로 배포 확인). 마이그레이션은 선택(가드 적용 시).
- **부수**: `main`에서 `feat/v1.5-roster-mgmt` 브랜치 분기 후 작업. `README.md`의 "관리자만 학생 추가/삭제" 문구 갱신. FE 편집마다 `check-web.mjs` 실행.
- **검증 한계(명시)**: 로컬 상한 = JSX 컴파일 + 로그인 화면 렌더. 인증 후 화면(명단탭/추가폼/새가족 삭제)은 이 환경에서 런타임 검증 불가 → 실기기 테스트 필요. 정직하게 보고.
