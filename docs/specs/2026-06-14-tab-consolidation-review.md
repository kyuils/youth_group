# 시트 탭 구성 검토 (#3 STUDENTS vs 2026 반편성 · #4 교사 탭 2개)

작성: 2026-06-14 · 근거: gas/ 코드 정적 분석

## #3 — `STUDENTS` 탭과 `2026 반편성` 탭, 중복인가?

### 결론: **실제 데이터 중복이 아니다.** `STUDENTS`는 `2026 반편성`의 파생 뷰다.

코드 근거:
- `gas/Sheet.gs`의 `SHEET_NAMES`에서 `STUDENTS:'STUDENTS'`, `RAW_CLASS:'2026 반편성'`로 둘 다 참조.
- 신규/등반 학생 쓰기는 **모두 `2026 반편성`(RAW_CLASS)에만** 일어난다
  (`appendNewcomerRow_`는 v1.4부터 새가족부, `copyNewcomerToRawSheet_`·`updateClassInRawSheet_`는 RAW_CLASS).
- 읽기는 `STUDENTS`를 `readTableSmart_(SHEET_NAMES.STUDENTS, '이름')`로 읽는다. `STUDENTS`는
  ArrayFormula 스필(spill) 뷰라 수만 개의 팬텀 빈 행이 생겨, 이를 회피하려 `readTableSmart_`가
  존재한다(주석 Sheet.gs). 즉 `STUDENTS`는 **입력 편의/표시용으로 `2026 반편성`을 가공한 읽기 전용 뷰**.

### 권고
1. **현 구조 유지가 합리적.** 같은 데이터를 두 번 입력하는 "중복"이 아니라, 원본(`2026 반편성`) +
   앱이 읽기 좋은 뷰(`STUDENTS`)의 1:1 파생 관계다. 운영자는 `2026 반편성`만 편집하면 된다.
2. 단, `STUDENTS`가 ArrayFormula 뷰라서 생기는 팬텀 행 비용이 있다. 더 단순화하려면:
   - (선택) 코드가 `STUDENTS` 대신 `2026 반편성`을 직접 읽도록 바꾸고 `STUDENTS` 탭을 폐지.
     이러면 탭이 1개로 줄지만, `2026 반편성`의 헤더가 1~2행 구조(코드 가정상)라 컬럼 매핑을
     재작성해야 하고, 실제 시트를 봐야 안전하다. **데이터 손실 위험이 있어 운영자가 직접 판단/수행 권장.**
3. **요약**: "둘 중 하나만 있으면 되지 않나?" → 기능상 `2026 반편성`만으로 충분하지만, `STUDENTS`
   뷰는 읽기 성능·팬텀 회피를 위한 의도적 설계물이다. 지금은 **유지**, 장기적으로 단순화하려면
   위 2번을 실제 시트 확인 후 진행.

## #4 — 교사 탭이 2개?

### 코드가 참조하는 교사 탭은 **`TEACHERS` 하나뿐이다.**
- `SHEET_NAMES.TEACHERS:'TEACHERS'` 단일. `lookupTeacher`(인증), `handleGraduate`의 교사 검증,
  `migration_v11_set_roles`가 모두 `TEACHERS`만 읽는다.
- `2026 반편성`의 B열 `교사명`은 교사 명단이 아니라 학생 행의 담임 표기(현재 앱은 P열 `반`을 사용).

### 두 번째 교사 탭의 정체는 코드만으로 알 수 없다 (앱이 사용하지 않음).
가능성: (a) 운영 초기의 레거시/백업, (b) 수기 관리용 별도 표, (c) 다른 용도.

### 결정 (2026-06-14, 사용자 확인): **두 번째 교사 탭은 파기.**
- 앱은 `TEACHERS` 탭만 참조하므로 **코드 변경 불필요.**
- 운영자는 구글 시트에서 두 번째 교사 탭을 **직접 삭제**하면 된다 (앱 동작 영향 없음).
- 삭제 전 권장: 그 탭에만 있는 정보가 없는지 한 번 확인 후 삭제.
