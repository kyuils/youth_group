# v1.3 — 관리자 통계: 전체·반별·학년별 출석 + 새가족·등반 현황 설계서

작성: 2026-06-13 · 상태: 승인됨 (사용자 확인)

## 1. 목표

관리자(`read_all` 권한)가 관리자 탭에서 다음을 볼 수 있게 한다:

1. **전체 출석 통계** — 올해 누적 + 주별 추이
2. **반별 통계** — 반 단위 출석률 비교
3. **학년별 통계** — 중1~고3 학년 단위 출석률 비교
4. **새가족 출석 현황** — 새가족 반 학생별 출석
5. **등반 진행 현황** — 교육(1~4주) 진행 중인 학생 + 올해 등반 완료자 명단

## 2. 현재 상태 (코드 확인됨)

- `AdminView`(web/index.html:2388)는 `getAllRosters`/`getAllAttendance`/`getAllPrayers`를 받아
  반별 출석 합계·명단·기도제목을 평면 나열한다. 학년별·추이·새가족·등반 정보 없음.
- `ROSTER_FIELDS = ['id','이름','학년','성별']`(gas/Actions.gs:9) — 학년별 집계에 필요한
  데이터는 이미 프론트에 내려온다.
- `getAllAttendance` row에는 `teacher`(기록 당시 반)와 `status`, `date`가 있다(Actions.gs:477-485).
- `getNewcomers`(Actions.gs:323)는 **현재 반='새가족'인 학생만** 반환한다(Sheet.gs:172-178).
  등반하면 반이 바뀌어 목록에서 사라지므로, 등반 완료자 명단은 현재 API로 조회 불가.
  단, NEWCOMER_PROGRESS 시트에는 등반일·등반반 기록이 남는다(Sheet.gs:340-).
- 프론트 캐시: `graduate` 액션이 `getNewcomers` 캐시를 무효화한다(web/index.html:703).

## 3. 구현 방식 결정

**클라이언트 집계 (채택)** — AdminView가 이미 받는 전체 데이터로 프론트에서 계산.
기존 StatsView/WeeklyView와 동일 패턴, 데이터 규모(수십 명 × 연 50여 회)가 작아 충분.

서버 집계(GAS가 통계 계산)는 백엔드 비대화 + 기존 패턴 불일치로 기각.

**GAS 수정은 1건만**: 등반 완료자 조회 (아래 §5).

## 4. 화면 구조 — 관리자 탭 3세그먼트 재편

WeeklyView의 `seg-bar` 패턴을 재사용한다.

```
관리자 탭
├─ [통계] (기본 세그먼트)
│   ├ 전체 요약 카드: 올해 평균 출석률 · 예배 N회 · 학생 N명 · 최근 예배 출석 인원/%
│   ├ 주별 추이: 최근 12주 주별 출석 인원 미니 막대 차트 (CSS만, 라이브러리 없음)
│   ├ 반별 통계: 반 · 인원 · 누적 출석률 progress bar (stat-bar-card 스타일 재사용)
│   └ 학년별 통계: 중1→중2→중3→고1→고2→고3→기타 순, 동일 형식
├─ [새가족·등반]
│   ├ 새가족 출석 현황: 새가족 학생별 올해 출석 횟수·출석률
│   ├ 등반 교육 진행: 학생별 1~4주 체크 상태 표시 (읽기 전용 — 수정은 새가족 탭)
│   └ 올해 등반 완료: 이름 · 등반일 · 등반반 명단 (올해 등반일 기준 필터)
└─ [명단·기도]: 기존 AdminView의 반 선택 + 학생 명단 + 기도제목 그대로 이동
```

데이터 로드: 기존 3개 호출에 `getNewcomers` 추가 (admin은 `read_all`로 호출 가능,
Actions.gs:326 확인됨). 모두 callApi 캐시(2분 TTL) 대상이라 비용 미미.

## 5. 백엔드 수정 (GAS 1건)

`handleGetNewcomers` 응답에 `graduates` 배열을 추가한다:

- NEWCOMER_PROGRESS 시트에서 `등반일`이 비어 있지 않은 행을 골라
  `loadStudentsMin_()`과 학생id로 조인:
  `{ id, '이름', '학년', graduatedOn(등반일), graduatedTo(등반반) }`
- 연도 필터는 하지 않고 전체 반환 — 올해 필터는 클라이언트에서 (클라 집계 패턴 일관).
- 학생 행이 STUDENTS에서 사라진 경우(비활성 등) 이름은 progress 행 기준 빈 값 허용,
  UI는 이름 없으면 `학생 {id}`로 표시.
- 권한: 기존 게이트 그대로 (`write_newcomer` 또는 `read_all`) — 새가족부 교사도
  자기가 등반 보낸 학생을 보는 것이 자연스러움.
- 새 액션 등록 없음 → Code.gs 라우팅 무변경. **GAS 재배포 1회 필요** (clasp push + 새 배포).

## 6. 집계 기준 (명확화)

| 항목 | 기준 |
|---|---|
| 기간 | 올해 고정 (기존 통계와 동일, `new Date().getFullYear()`) |
| 출석 판정 | `PRESENT_CODES`(P2/P3/P4) + 기타(ETC)는 기존 StatsView와 동일하게 **출석에 불포함** |
| 반별 분류 | **기록 당시의 반**(`row.teacher`) 기준 — 등반 학생의 새가족 시절 기록이 새가족 반으로 정확히 잡힘 |
| 학년별 분류 | **현재 명단의 학년** 기준 (출석 기록에 학년 없음). 학년 빈 값 → '기타' |
| 학년 정렬 | 중1 < 중2 < 중3 < 고1 < 고2 < 고3 < 기타 (고정 순서 배열) |
| 출석률 분모 | 현재 인원 × 올해 예배 횟수 (현행 AdminView 방식 유지) |
| 주별 추이 | 출석 기록의 date(주일)별 출석 인원 집계, 최근 12개 주일만 표시 |

### 알려진 한계 (문서화로 수용)

- 연중 전입·등반으로 반이 바뀐 학생은 분모(현재 인원) 기준이라 반별 출석률이 약간 왜곡될 수 있다.
- 반별 분류는 기록 당시 반, 분모 인원은 현재 반이라 등반 직후 새가족 반 출석률이 100%를 넘을 수 있다
  → 표시 시 100%로 캡.
- 올해 연도 고정은 v1.2와 동일한 기존 제약 (연초 작년 조회 불가 — 별도 백로그).
